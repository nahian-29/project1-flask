S= Single Responsibility Principle: Every class is responsible for exactly one thing. That means it has only one reason to change. If you change anything in that class, it will effect only one particular behavior of the software. That makes the code more robust because there will be less side effects. If a class had two responsibilities and thr programmer changed anything, the risk would be high that the programmer could also break the logic for the second behavior.
"A class should have one, and only one, reason to change."

O= Open-Closed Principle: The classes you use should be open for extension but closed for modification. This can be achieved with inheritance. You donâ€™t have to touch the class you want to extend if you create a subclass of it. The original class is closed for modification but you can add custom code to your subclass to add new behavior. Inheritance may be the most popular way to implement the Open-Closed Principle but it is not the only one. Objective-C, for example, offers categories which can be used to add methods to a class, without touching the original class and without subclassing it.
"You should be able to extend a classes behavior, without modifying it."

L= Liskov Substitution Principle: Assume we have a class B which is a subclass of A. If the program works with an object a of class A one can replace it with an object b of class B without changing the behavior of the program. The implementation of this principle can be a little bit tricky if one combines it with the Open-Closed Principle even if the behavior is extended of the class in a subclass, one must make sure that it still exchanges the base class with the derived class without breaking anything.
"Derived classes must be substitutable for their base classes."

I= Interface Segregation Principle: Classes should be as specialized as possible. You do not want any god classes that contain the whole application logic. The source code should be modular and every class should contain only the minimum necessary logic to achieve the desired behavior. The same goes for interfaces. Make small and specific interfaces so the client who implements them does not depend on methods it does not need. Instead of one class that can handle three special cases it is better to have three classes, one for each special case.
"Make fine grained interfaces that are client specific."

D= Dependency Inversion Principle: Classes should not depend on concrete details of other classes. Even classes from a high domain level should not handle the specific details of components from a lower level. Both low and high level classes should depend on the same abstractions. To create specific behavior one can use techniques like inheritance or interfaces.
"Depend on abstractions, not on concretions."
