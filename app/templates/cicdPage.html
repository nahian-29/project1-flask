Before explaining what Continuous Integration / Continuous Delivery (CI/CD) means, I would like to give a scenario:

Imagine going to work, and you had a team where the job was to take a space and design that room as if it was a place for students. Every member has their own vision but the issue is that the team has to work on one project file. Team members can download and work on their individual assignments but then the project would have multiple different files when only one is needed. Then, only committing to one file will result in having people redo their parts in one single file or having to wait on someone to finish their part in order to work on yours. It becomes a tense task to constantly having to wait on someone, see what was changed, then add your own work to the project, check if it doesn't crash or makes sense, then upload for the next person.

That's the essence of Continuous Integration (CI), there is one master copy that everyone involved would download, work on their part then merge that to the development branch or the live production branch. The master copy would essentially stay as is in case, somewhere along the line there's data loss but the two branches are the ones constantly changing through team members.

Continuous Delivery (CD) is constantly uploading to the branches to get constant and instant feedback or pushing the development version of the service to the live production state with ease. The reason that works with relatively small issues is that the team would know something went wrong in the development stage and don't have to panic and rush to fix something for live production.

There is a bit of automation involved in the deployment or to better put it, Actions that the deployment process would do and GitHub, in this case, will perform them as told. This application uses 2 actions per workflow, test runs and the deployment location (development will go to Docker*, live would go to the hosting site: Heroku).
You can see what actions are being done if you go to the Actions tab in the GitHub repository. In the dev version of this project, you'll see the test that I mentioned in the page before.

This project is done locally on my PC using PyCharm Professional. Then, once I'm done with any changes regarding the web application functions, I'll push the changes to the respective branch. When I'm sastified with the functionality with the project, I can push and merge the changes to the live production branch (the live server) and change the visual aspects since the functionality aspect of the project is not changing at all.

These are the two servers:

  • Live (This current one you are seeing): https://kaa46live.herokuapp.com
  • Dev: https://kaa46dev.herokuapp.com

There's also a cool part of this project as well, whenever the live or master branch is updated, the Dockerhub image is going to update as well with the new changes that you pushed through. It's all done through Actions, in this case our GitHub workflow files from earlier!
